# PRD 0002: FE Requirements

## Overview
This document defines the product requirements for the user interface of the sward management application for farmers. The primary goal of the FE is to facilitate compliance, optimization, and environmental safety regarding sward application.

## Key Features

1. **Sward Management Dashboard**
   - Provide an overview of current sward storage levels.
   - Display planned and historical application events.

2. **Compliance Tracking**
   - Ensure sward application meets local regulations.
   - Alert users of prohibited application periods or conditions.

3. **Optimization Engine**
   - Suggest optimal times and locations for sward application based on crop needs and nutrient content.
   - Calculate application rates to maximize yield and minimize waste.

4. **Weather Integration**
   - Incorporate real-time and forecasted weather data.
   - Prevent application scheduling during heavy rain or adverse weather conditions that could lead to runoff.

5. **Topology & Field Mapping**
   - Visualize field topology, slopes, and vulnerable zones.
   - Highlight areas with high risk of runoff.

6. **Waterway Protection**
   - Buffer zones near waterways must be clearly marked.
   - System must strictly prevent scheduling application within buffer zones to avoid runoff into waterways.

7. **User Profile & Farm Management**
   - Interfaces to set up and manage user profiles.
   - Interfaces to manage multiple farms and define fields within each farm.

8. **Event Tracking Interface**
   - Forms to record agricultural events (e.g., planting, fertiliser application, sward application, spraying, harvesting, tilling) for individual fields.

9. **Inventory & Equipment FE**
   - Interfaces to track sward storage capacities, chemical/pesticide inventory (including MAPP numbers), and equipment details (e.g., LESSE tracking).

10. **Reporting & Export**
    - Views to generate and export compliance reports for authorities (e.g., DAERA/NIEA).
    - Features to export digital pesticide records, annual fertilisation accounts, and general farm records.

## Design & UX Patterns

### 1. Premium Aesthetic
All user-facing modules must adhere to the "FieldMetric" premium design system:
- **Typography**: Primary use of 'Work Sans' or similar modern sans-serif fonts from Google Fonts.
- **Color Palette**: Use curated HSL colors for depth and visual interest (e.g., `#154212` for primary, `#faf9f5` for surface).
- **Material Symbols**: Use Material Symbols Outlined for all iconography with consistent weight/fill variations.

### 2. Layout Patterns
- **Bento Grid**: Use Bento-style grids for dashboards and statistical overviews to provide a modern, organized information hierarchy.
- **Card-Based FE**: Employ minimalist cards with subtle shadows and rounded corners (at least `xl` or `2xl`) for list items and record details.
- **Timeline View**: Activity logs must use a high-contrast timeline layout with specific iconography for different agricultural events.

### 3. Navigation & Interaction
- **Main Layout Shell**: A persistent `MainLayoutComponent` must house the `TopAppBar` and `BottomNavBar`. Feature views are rendered within a nested `<router-outlet>` to prevent layout jitter during navigation.
- **Micro-Animations**: Implement subtle interactions such as `scale-95` on active states, hover transitions, and smooth transitions between route states.
- **Fixed Bottom Nav**: A glassmorphism-style bottom navigation bar with `backdrop-blur` for quick access to core modules (Dashboard, Farms/Fields, Events, Profile).

## Technical Requirements
- As per PRD 0001, the fe will be built using Angular, utilizing Tailwind CSS for styling and layout.
- The FE must be responsive and accessible, specifically optimized for high-contrast visibility in outdoor agricultural environments.
- The application must support a "Main Layout" pattern to reduce redundant rendering of navigation elements.

## Runtime Configuration & Microfrontends

### 1. Dynamic Configuration Loading
The application must load its configuration from `/assets/contents/shell-config.json` before bootstrapping the Angular environment. This configuration is used to:
- Initialize Microfrontend federation via Native Federation.
- Define available remotes and their locations.
- Configure dynamic routing and navigation menu items.
- Provide OpenTelemetry (OTel) settings.

### 2. Configuration Schema
The configuration file should follow a structured schema:
- `remotes`: A record of microfrontend names and their URLs.
- `mfeRoutes`: A list of route configurations including path, display name, remote name, exposed module, and component name.
- `menu`: A list of menu items with name, route, and optional icon.
- `otel`: (Optional) OpenTelemetry configuration including `collectorUrl` and `logLevel`.

### 3. Application Integration
- The loaded configuration must be available throughout the application via an `InjectionToken` named `MFE_CONFIG`.
- The `APP_INITIALIZER` pattern should be used where necessary to ensure dependent services (like Auth) are initialized with the loaded configuration.
- Routing should be dynamically generated based on the `mfeRoutes` provided in the configuration.
