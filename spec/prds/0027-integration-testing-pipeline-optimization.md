# PRD 0027: Integration Testing Pipeline Optimization

## Context & Motivation

Our current CI/CD integration testing pipeline deploys test reports (generated by Robot Framework) to GitHub Pages. However, the checkout of the `gh-pages` branch is extremely slow because it performs a full checkout of the branch's historical data, which contains all past test runs and artifacts.

Furthermore, the checkout, preparation, and directory pruning for `gh-pages` are executed sequentially *after* the integration tests have finished running. Since these tasks only depend on the existing state of the `gh-pages` branch (and not on the new test results), they can be decoupled and run in parallel/earlier in the workflow to minimize post-test processing times.

## Requirements

### 1. Shallow Clone the gh-pages Branch
- **Requirement**: The checkout action for the `gh-pages` branch must be configured with a depth of `1` (`fetch-depth: 1`).
- **Rationale**: We only need the current state of the pages branch to merge new results and prune old ones; we do not need its git commit history.

### 2. Parallelize Checkout and Pruning Workflow Steps
- **Requirement**: Move the step `Checkout gh-pages` and the pruning logic of old PR reports so they run before or concurrently with the test execution.
- **Implementation Detail**:
  - Perform the shallow checkout of `gh-pages` and preparation of the staging directory (`./gh-pages-publish`) early in the pipeline (e.g., right before `Wait for Docker Images`).
  - Prune old PRs in the staging directory early.
  - After the tests finish, copy only the new test results into the staging directory and deploy the updated files.
